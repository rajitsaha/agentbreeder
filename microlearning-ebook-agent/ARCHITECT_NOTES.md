# ARCHITECT_NOTES — microlearning-ebook-agent
Generated by /agent-build on 2026-04-28. Edit freely — this file is for humans.

## Business Goal
Help corporate L&D teams produce bite-sized training material on demand without hiring instructional designers.

## Technical Use Case
User submits a topic → agent researches it across the web → agent extracts key concepts → agent structures content into microlearning modules (intro, lesson, quiz, summary) → agent generates the ebook as PDF/EPUB → returns a downloadable file.

## Why Google ADK (Full Code)?
Four of four state complexity flags selected (loops/retries, checkpoints, HITL, parallel research). That is squarely Full Code territory — Low Code YAML cannot express resume-from-failure or wait-for-human-approval semantics. Google ADK was preferred over LangGraph because the deploy target is GCP Cloud Run, keeping IAM, billing, and managed services in one cloud. ADK's native session+memory backends fit the checkpoint requirement without bolting on third-party state stores.

## Why gemini-2.5-pro?
The pipeline is research synthesis → pedagogical structuring → quiz generation. All three steps reward strong reasoning, especially "quiz with non-trivial distractors" and "surface disagreement when sources conflict." Gemini 2.5 Pro is GCP-native (one cloud, one bill, one IAM) and benchmarks at the level needed for instructional-designer-grade output. Cheaper Gemini Flash variants risk shallow quizzes and missed citations.

## Why No RAG?
DATA_FLAGS = (d) only — live web research. There is no internal corpus to index, and the value of "research it across the web" is freshness. A vector index would just stale the output. Web search runs as a tool call (research_web) on every run.

## Why Long-term (PostgreSQL)?
The agent has two state-persistence needs that outlast a single function invocation: ADK session checkpoints (resume-from-failure on long ebook generations, possibly minutes per run) and HITL approval state (an outline can sit pending for hours waiting on an L&D reviewer). Both demand durable storage. Redis was rejected — short-term per-session context is not the bottleneck; durability is.

## Why MCP servers?
Web research is the only external data dependency, and the MCP ecosystem already has well-tested servers (Tavily, fetch, etc.). Building a bespoke search client would duplicate work and skip MCP's auth/quota handling. A2A was not added — the parallel research branches are internal parallelism inside one agent, not delegation to specialized sub-agents.

## Why GCP Cloud Run?
CLOUD_PREFERENCE = GCP. SCALE_PROFILE = (d) low-volume internal tooling. Cloud Run scales to zero between ebook generations, so a team that runs five ebooks a week pays cents. Long-running generations (multi-minute) fit Cloud Run's 60-min request budget. GKE was overkill, App Engine has worse cold-start ergonomics for this workload.

## Why These Eval Dimensions?
- **Citation accuracy** — The whole product promise is "trustworthy training material." A miscite invalidates the ebook for L&D use.
- **Hallucination rate** — Tied to citation accuracy. Generated stats with no source destroy reviewer trust on first encounter.
- **Content completeness** — L&D teams expect a known structure (intro, lessons, quizzes, summary). Missing sections fail the deliverable contract.
- **Pedagogical structure quality** — "Without hiring instructional designers" only holds if the output reflects ID craft (Bloom's progression, stated outcomes, recap loops).
- **Quiz answer correctness** — Quizzes are the part most likely to ship to learners verbatim. A wrong-answer quiz erodes learner trust faster than a weak lesson.
